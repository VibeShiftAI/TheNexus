012: # Plan: ChoreTracker - Weekly Household Task Manager (v1)
013: 
014: # ChoreTracker Project Plan
015: 
016: This project aims to build a local-first web application to track household chores. The system calculates upcoming chores for the week based on frequency and history (who performed the task last).
017: 
018: ## Tech Stack
019: - **Framework**: Next.js (App Router)
020: - **Styling**: Tailwind CSS + Shadcn UI
021: - **Database**: SQLite with Drizzle ORM (Local file-based storage)
022: - **Icons**: Lucide-react
023: 
024: ---
025: 
026: ### Task 1: Project Initialization & UI Foundation
027: - **Workflow**: Nexus Prime
028: - **Goal**: Initialize the Next.js project with Tailwind CSS and install necessary UI components.
029: - **Context & Execution**: Create a new Next.js project. Install Shadcn UI and set up a basic layout with a sidebar for navigation (Dashboard, Chores, Household). Use `lucide-react` for icons.
030: - **Acceptance Criteria**:
031:   - Next.js app is running on localhost:3000.
032:   - Basic layout with navigation is visible.
033:   - Tailwind and Shadcn are configured correctly.
034: 
035: ### Task 2: Database Schema & Drizzle Setup
036: - **Workflow**: Nexus Prime
037: - **Goal**: Define the data model for users, chores, and chore history using SQLite.
038: - **Context & Execution**: Setup Drizzle ORM with `better-sqlite3`. Define the following tables:
039:   - `users`: id, name, avatar_url
040:   - `chores`: id, name, description, frequency_days (integer)
041:   - `chore_completions`: id, chore_id, user_id, completed_at
042: - **Acceptance Criteria**:
043:   - `schema.ts` file exists with defined relations.
044:   - Database migrations can be generated and applied.
045:   - A local `.db` file is initialized.
046: 
047: ### Task 3: Household Management (Users)
048: - **Workflow**: Nexus Prime
049: - **Goal**: Create the interface to add and manage household members.
050: - **Context & Execution**: Build a "Household" page with a list of users and a form/dialog to add a new user. Store users in the `users` table.
051: - **Acceptance Criteria**:
052:   - User can add a household member by name.
053:   - List of members displays accurately from the database.
054: 
055: ### Task 4: Chore Definition Management
056: - **Workflow**: Nexus Prime
057: - **Goal**: Create the interface to define chores and their frequencies.
058: - **Context & Execution**: Build a "Chores" page. Each chore needs a name and a frequency in days (e.g., 'Vacuuming' every 7 days). Implement CRUD operations using Server Actions.
059: - **Acceptance Criteria**:
060:   - User can create, edit, and delete chores.
061:   - Frequency is stored as an integer representing days.
062: 
063: ### Task 5: Chore Engine & Due Date Logic
064: - **Workflow**: Nexus Prime
065: - **Goal**: Implement the logic to determine when a chore is due and who should do it.
066: - **Context & Execution**: Create a utility function `getWeeklySummary()` that:
067:   1. Looks at the last completion date for each chore.
068:   2. Calculates the next due date based on frequency.
069:   3. If due within the next 7 days, includes it in the summary.
070:   4. Suggests the next assignee based on who performed it least recently among all users.
071: - **Acceptance Criteria**:
072:   - Function correctly identifies chores due in the current week.
073:   - Assignment logic correctly picks the user with the oldest completion timestamp for that specific chore.
074: 
075: ### Task 6: Weekly Dashboard UI
076: - **Workflow**: Nexus Prime
077: - **Goal**: Build the main dashboard showing the "Weekly Summary".
078: - **Context & Execution**: Create a dashboard view that displays cards for each chore due this week. Each card should show: Chore Name, Due Date, and "Suggested Assignee". Include a "Mark as Done" button.
079: - **Acceptance Criteria**:
080:   - Dashboard renders a list of tasks for the upcoming week.
081:   - "Mark as Done" button triggers a server action that logs a new completion.
082: 
083: ### Task 7: Completion Logging & History
084: - **Workflow**: Nexus Prime
085: - **Goal**: Finalize the feedback loop by recording task completions.
086: - **Context & Execution**: When a user marks a chore as done, prompt them to select which user actually did it (default to the suggested user). Save this to `chore_completions`. Refresh the dashboard state immediately.
087: - **Acceptance Criteria**:
088:   - Clicking "Mark as Done" adds a record to the database.
089:   - The dashboard updates, moving the chore to the bottom or removing it from the "Due" list for the remainder of its cycle.
